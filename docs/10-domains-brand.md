# Domains and Brand

Working name: Status.

The name is simple, direct, and accurate. It may be difficult to own because it is generic, so this document keeps naming flexible.

## Naming criteria

The name should feel:

- native;
- calm;
- operational;
- trustworthy;
- short;
- easy to say;
- not too enterprise;
- not too playful;
- not too AI-coded.

The name should support:

- macOS app;
- iOS app;
- plugin store;
- optional cloud relay;
- developer documentation;
- future marketplace.

## Working brand options

### Status

Pros:

- immediately understandable;
- product explains itself;
- perfect for event/status dashboard;
- good app name if available enough.

Cons:

- generic;
- likely hard to own domains/search;
- possible App Store search ambiguity.

### State

Pros:

- short;
- technical but clean;
- works with events/resources;
- less literal than Status.

Cons:

- still generic;
- could sound abstract.

### Desk

Pros:

- simple;
- operational;
- works for a personal cockpit;
- not overly technical.

Cons:

- too broad;
- less event/status specific.

### Beacon

Pros:

- implies signals/attention;
- warmer brand;
- works with alerts.

Cons:

- common;
- can sound monitoring-heavy.

### Overzicht

Pros:

- distinctive;
- means overview;
- personal flavour;
- likely more ownable.

Cons:

- Dutch word may be hard outside NL/BE;
- spelling/pronunciation friction.

### Signal Desk

Pros:

- explains the category;
- good as product descriptor.

Cons:

- less elegant;
- longer.

## Recommended naming approach

Keep `Status` as the working product name until the product shape is validated.

Use a descriptive subtitle:

```txt
Status
A native status layer for everything you run.
```

If the final brand cannot own `Status`, keep it as a product category term and choose a more ownable brand later.

## Product URLs

Possible structure:

```txt
status.app
status.app/download
status.app/plugins
status.app/docs
status.app/developers
plugins.status.app
docs.status.app
hooks.status.app
api.status.app
```

For an ownable alternative:

```txt
statuskit.app
plugins.statuskit.app
hooks.statuskit.app
docs.statuskit.app
```

## Plugin registry domains

Possible:

```txt
plugins.status.app/index.json
plugins.status.app/plugins/{pluginId}/{version}/plugin.zip
```

or:

```txt
registry.status.app/index.json
```

`plugins.` is clearer for humans. `registry.` is clearer for tooling.

## Cloud relay domains

Possible:

```txt
hooks.status.app/{userId}/{pluginId}/{token}
relay.status.app
```

`hooks.` is the clearest domain for incoming webhooks.

## Visual direction

Status should feel like:

- native Mac utility;
- calm dashboard;
- operational cockpit;
- warm-neutral background;
- restrained accent colours;
- clear severity states;
- compact cards;
- polished but not decorative.

Avoid:

- dark cyber dashboard aesthetic;
- enterprise DevOps wallboard;
- generic SaaS analytics grid;
- colourful toy-like widget app;
- AI assistant branding.

## Colour language

Severity should be consistent:

```txt
ok       → green
notice   → blue or neutral
warning  → amber
action   → purple/blue
critical → red/orange
```

Use colour for meaning, not decoration.

## Voice

Status should speak plainly.

Good:

```txt
Tiko Yes No was rejected.
Reviewer could not complete login.
Open App Store Connect.
```

Bad:

```txt
Oops! Something went wrong with your awesome app submission.
```

Good:

```txt
Views are down 18% compared with the previous 28 days.
```

Bad:

```txt
Your channel performance has dramatically shifted.
```
